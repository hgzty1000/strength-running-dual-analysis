# 低保真 UE 设计

- 日期: 2026-07-08
- 状态: 草案
- 上游文档: [Demo 范围与信息架构设计](./demo-scope-and-information-architecture.md)
- 用途: 在进入 UI 视觉与技术架构前,先固定 demo 第一版的页面结构、信息层级、核心交互与占位策略。

---

## 1. 设计目标

低保真 UE 的目标不是画最终视觉稿,而是回答:

1. 用户打开平台后看到什么。
2. 各一级模块承载什么信息。
3. demo P0 能力在哪些页面点亮。
4. P1/未来能力在哪里预留,避免后续大返工。
5. AI 分析、目标配置、数据导入、报告复盘之间的页面边界如何保持清楚。

核心原则:

> 页面骨架按完整版产品设计,功能实现按 demo P0 点亮。

## 2. 全局导航

### 2.1 导航实现原则

Demo 采用**一套页面、一套路由、一套组件**。桌面端和手机端只改变导航外壳与局部排布,不做两套页面。

核心原因:后续更新和 debug 必须做到一次改动,桌面和手机浏览器同时生效,避免 PC/Mobile 两套 UI 逻辑分叉。

主导航配置只有一份:

```text
首页
数据
目标
分析
报告
设置
```

owner/admin 入口隐藏或仅 owner 可见:

```text
管理
```

### 2.2 桌面端导航

桌面端渲染为左侧侧边栏:

```text
┌──────────────┬──────────────────────┐
│ 双修运动平台  │ 页面标题 / 页面操作    │
├──────────────┼──────────────────────┤
│ 首页          │                      │
│ 数据          │                      │
│ 目标          │      页面内容         │
│ 分析          │                      │
│ 报告          │                      │
│ 设置          │                      │
│ 管理(owner)   │                      │
└──────────────┴──────────────────────┘
```

### 2.3 手机端导航

手机端不单独设计底部 Tab,避免和桌面端信息架构分叉。手机端使用顶部栏 + 汉堡菜单:

```text
┌──────────────────────┐
│ ☰  页面标题           │
├──────────────────────┤
│ 页面内容              │
└──────────────────────┘
```

点击 `☰` 展开同一份主导航:

```text
首页
数据
目标
分析
报告
设置
```

这样目标仍是一级模块,不需要额外做「我的」聚合页,也不需要维护移动端专属导航逻辑。

### 2.4 响应式页面规则

- 同一页面组件在宽屏上可多列排布,窄屏上变单列。
- 列表优先使用卡片式设计,减少桌面表格/手机卡片两套实现。
- 报告页使用统一区块卡片,桌面和手机只改变宽度与折叠策略。
- 二级页面在桌面内容区内显示返回/面包屑;手机端顶部显示返回。
- 不创建 `DesktopXxxPage` / `MobileXxxPage` 两套页面。

### 2.5 UI 实现原则

- 不使用大型 UI 组件库,不引入 Ant Design、Material UI、Element、Naive UI、Bootstrap 组件库等大而全设计系统。
- 页面组件、布局、样式和响应式行为由项目自写轻量组件控制。
- 可使用必要基础工具库,但 UI 层尽量直接实现。
- 基础组件包括:Button、Input、Select、Textarea、Card、Tabs、Alert、Toast、FileInput、ActivityCard、ReportSection、FormField 等。
- 样式优先使用普通 CSS / CSS Modules / CSS variables。
- 目标是降低依赖、方便 debug,并保证一次改动桌面和手机同时生效。

### 2.6 全局页面规则

- Key、用户档案、账号安全统一放「设置」,不放首页。
- 数据导入统一放「数据」,不和分析页混在一起。
- 目标配置独立成「目标」,不塞进分析页。
- 分析页负责触发 AI 分析。
- 报告页负责查看和复盘结果。
- P1 功能可以隐藏或占位,但要知道未来位置。

#### 5.4.3 demo 点亮

- ZIP 上传。
- FIT 解析。
- 查重。
- 活动列表。

- 场景识别:
  - 户外路跑。
  - 体育场/操场跑。

- 地图展示。

- Garmin 自动同步。
- 非跑步专项分析。
- 文件批量管理高级功能。

- ZIP 内无 FIT 或多 FIT 时,给明确错误/选择策略。
- 气温、功率等字段缺失不算失败,只标记字段缺失。
- 气温缺失时,后续可通过户外活动 GPS + 时间查第三方历史天气补全;该能力不作为 demo 内容,优先级低,也不阻塞 FIT 导入。
- 非跑步 FIT 可入库为通用 Garmin 活动,但标记“暂未纳入深度分析”。

## 7. 分析页

### 7.1 页面目标

分析页负责触发 AI 分析,并在触发前让用户知道本次分析会使用什么数据和目标。

### 7.2 低保真结构

```text
┌────────────────────────────────────────────┐
│ 分析                                       │
├────────────────────────────────────────────┤
│ 分析范围                                   │
│ 默认: 最近 6 周                            │
│ [开始日期] [结束日期]                      │
│                                            │
│ 使用目标                                   │
│ 当前目标: 跑步比赛优先 + 力量维持底线        │
│ 目标版本: v3                               │
│ [查看目标]                                 │
│                                            │
│ 数据覆盖检查                               │
│ 力量: xx 天 / xx 组                         │
│ 跑步: xx 次 / xx km                         │
│ Garmin 字段缺失: 气温缺失                   │
│ 待补肌群动作: xx                            │
│                                            │
│ 休整标注                                   │
│ 当前范围内有 1 条休整标注                   │
│ [新增/查看休整标注]                         │
│                                            │
│ 分析前提示                                 │
│ 本次分析会调用云端 LLM,发送必要训练上下文。  │
│                                            │
│ [生成 AI 分析]                              │
└────────────────────────────────────────────┘
```

### 7.3 demo 点亮

- 默认最近 6 周。
- 可选日期范围。
- 使用当前目标配置。
- 展示数据覆盖摘要。
- 展示休整标注摘要。
- 主动触发分析。
- 分析完成后跳转报告详情。

### 7.4 demo 不做

- 多种分析模板。
- 多模型比较。
- 定时分析。
- 后台自动分析。
- 外部 API 触发分析。

### 7.5 交互规则

- 没有 LLM Key 时,生成分析按钮不可用。
- 没有目标配置时,提示先去目标页配置。
- 数据覆盖不足时不一定禁止分析,但必须提示置信度会受影响。
- 分析失败要保留错误信息,不生成半截报告。
- 重新分析生成新报告,不覆盖旧报告。

## 8. 休整标注入口

### 8.2 低保真结构

```text

- 日期范围: [开始] [结束]
- 影响范围: 跑步 / 力量 / 腿部 / 全部 / 自定义
- 备注: [________________________]
[保存]
```

### 8.3 demo 点亮

- 新增。
- 列表查看。
- 用于分析上下文。

### 8.4 demo 不做

- tag 管理。
- 复杂分类。
- 自动识别休整。

### 8.5 交互规则

- 休整标注解释过去/某段事实。
- 当前/未来判断尺仍归目标配置。
- 同一事件若同时影响过去解释和未来判断,应拆成休整标注 + 目标配置两个对象。

## 12. Demo 占位 / 隐藏策略

### 12.1 建议隐藏的能力

这些功能 demo 不出现,避免干扰主路径:

- 手动记录表单。
- 平台 API Key 管理。
- 外部 agent 上下文包。
- 邀请码管理。
- 用户管理。

### 12.2 建议占位的能力

这些功能和当前页面关系强,可以占位说明:

- AI 目标澄清:放目标页。
- AI 浅追问:放报告页或分析页。
- 营养素建议:可在报告建议区标记后续支持。
- 动作肌群人工修正:可在数据/管理中后续支持。

### 12.3 占位文案原则

- 不说“不可用”这种负面文案。
- 用“后续支持 / demo 阶段暂不开放”。
- 明确原因时要短:
  - “为避免目标与分析混淆,该能力后续独立开放。”
  - “demo 阶段先验证双线分析核心闭环。”

## 13. 核心交互路径

### 13.1 首次使用

```text
登录
→ 首页空状态提示
→ 设置用户档案
→ 设置训记 Key / LLM Key
→ 数据 / 训记 / 同步
→ 数据 / Garmin / 上传 ZIP
→ 目标 / 填当前目标配置
→ 分析 / 检查数据覆盖
→ 生成分析
→ 报告 / 查看结果
```

### 13.2 日常使用

```text
登录
→ 首页查看状态
→ 同步训记
→ 上传新 Garmin
→ 如目标变化则更新目标
→ 分析
→ 查看报告
```

### 13.3 报告误判后重新分析

```text
查看报告
→ 发现误判
→ 分析页新增休整标注 / 目标页更新目标 / 数据页补上传
→ 重新分析
→ 新报告保存
→ 旧报告保留
```

### 13.4 目标变化

```text
目标页
→ 编辑当前目标
→ 保存为新版本
→ 后续报告引用新目标版本
→ 历史报告仍引用旧目标版本
```
