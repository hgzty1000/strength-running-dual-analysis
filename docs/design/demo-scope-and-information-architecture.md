# Demo 范围与信息架构设计

- 日期: 2026-07-08
- 状态: 草案
- 用途: 把已收敛的需求转成 demo 产品范围、信息架构与页面骨架,作为进入 UE/UI 与技术架构前的设计基准。

---

## 1. Demo 核心验证假设

Demo 第一版不是完整产品,而是验证一个核心假设:

> 平台能否在单用户场景下,基于训记力量数据、Garmin 跑步 FIT、当前目标配置和极简休整标注,生成可存档的双线训练分析报告,识别冲突/过量风险并给出方向性建议。

如果这个假设成立,项目值得继续投入;如果这个假设不成立,其他功能做得再完整也无法证明平台价值。

核心闭环:

```text
训记力量数据
+ Garmin 跑步活动
+ 当前目标配置
+ 必要休整标注
→ AI 双线分析报告
→ 冲突/过量判断
→ 短周期方向性建议
→ 历史报告复盘
```

## 2. Demo P0 / P1 / 明确不做

### 2.1 P0 必须做

P0 只包含验证核心价值所必需的能力。

1. **预置 owner 账号 + 登录**
   - demo 阶段只有平台所有者本人使用。
   - 数据模型仍从第一天按多用户建模,所有数据带 `user_id`。
   - 不开放公开注册。

2. **凭证管理:训记 Key + LLM Key**
   - 每用户一份训记 Key,加密存储,不日志打印,不完整回显。
   - 每用户一份 LLM Key,由平台所有者配置,加密存储。
   - LLM provider/base URL/model 为平台级配置。
   - 没有 LLM Key 的用户不能触发 AI 分析。

3. **训记同步**
   - 手动触发。
   - 首次全量。
   - 后续增量:从上次同步日当天到今天。
   - 指定日重拉。
   - 本地 SQLite 缓存。
   - 训记镜像只读,不写回。

4. **动作 → 肌群映射基础能力**
   - 训记 catalog 有 `type` 的直接用。
   - 缺失肌群的动作进入待补列表。
   - AI 补全一次后固化保存。
   - 后续复用。
   - 记录来源: `xunji_type` / `ai_inferred` / `user_corrected` 预留。
   - demo 不做复杂人工管理 UI。

5. **Garmin ZIP/FIT 上传与解析**
   - 上传 Garmin zip。
   - 解压出 `.fit`。
   - 解析通用 Garmin 活动。
   - 查重。
   - 标准化 `activity_family` / `activity_variant`。
   - running family 做完整指标提取。
   - 支持已验证场景:户外路跑、体育场/操场跑、跑步机。
   - 非跑步 FIT 只做通用解析、查重、存档、类型标记和元数据展示,暂不进入深度分析。

6. **当前目标配置(手动维护)**
   - 手动编辑目标配置。
   - 保存版本。
   - 字段最小包括:主目标、跑步目标说明、力量底线说明、冲突取舍说明、生效日期、备注/不确定项。
   - AI 目标澄清暂不实现,只预留入口。

7. **极简休整标注**
   - 日期范围。
   - 影响范围:跑步 / 力量 / 腿部 / 全部 / 自定义。
   - 备注。
   - 用于解释异常缺训或恢复期。
   - tag、管理与复杂联动后置。

8. **AI 分析报告**
   - 用户主动触发。
   - 使用当前目标配置版本。
   - 读取训记/Garmin/休整标注/肌群映射/用户档案。
   - 生成结构化报告 + 叙述层。
   - 必须声明分析前提、数据覆盖、关键不确定项与置信度。
   - 不生成完整训练计划,不提供精确训练处方,不替代医疗/康复判断。

9. **方向性建议**
   - 加 / 减 / 维持方向。
   - 排布约束。
   - 优先级取舍。
   - 观察指标。
   - 不排具体日历课表。

10. **历史报告保存**
    - 每次报告保存。
    - 历史报告列表。
    - 查看报告详情。
    - 旧报告不自动改。
    - 重新分析生成新报告。

11. **用户档案**
    - 身高、体重、年龄、性别。
    - 只存当前值。
    - 不存目标。
    - 不追踪体重趋势。

## 3. 部署与注册边界

### 3.1 部署策略

Demo 分两步:

1. **本地开发 / 本地验证**
   - 先在本地跑通核心闭环。
   - 用真实训记数据与 Garmin FIT 样本验证分析价值。

2. **云服务器方案 A 部署**
   - 跑通后部署到云服务器。
   - 仍按 ADR 0001 的方案 A: `http://<公网IP>:<端口>` IP 直连。
   - 不绑域名、不备案、不做正规 HTTPS。
   - 仅供平台所有者本人 / 可信设备使用。

### 3.2 注册策略

- demo P0 使用**预置 owner 账号**。
- 默认关闭公开注册。
- 如后续开放注册,必须使用邀请码。
- 邀请码只控制谁能注册账号,不等于允许朋友填训记 Key。
- 朋友输入训记 Key 必须等方案 C(正规 HTTPS)就位。
- 新用户默认没有 LLM Key;没有 LLM Key 不能触发 AI 分析。

### 3.3 上传安全边界

Garmin 上传在技术设计中必须考虑:

- 限制文件大小。
- 只接受 `.zip`。
- zip 内只处理 `.fit`。
- 限制 zip 内文件数量。
- 防 zip slip 路径穿越。
- 不解压到可执行目录。
- 解析失败安全报错。
- 原始文件按 `user_id` 隔离存储。

## 4. 完整版信息架构

页面骨架按未来完整产品设计,功能实现按 demo 范围点亮。

主导航:

```text
首页
数据
目标
分析
报告
设置
```

owner/admin 隐藏模块:

```text
管理
```

设计原则:

- 页面骨架按完整版设计。
- demo 只点亮最小闭环能力。
- 目标模块独立,不混入分析。
- 数据模块按数据源拆分,不混成一个导入按钮。
- 分析负责触发,报告负责复盘。
- 设置负责档案和凭证,不把 Key 塞首页。
- 管理能力预留 owner/admin,不影响普通用户路径。

| 数据-训记 | Key、同步、重拉、日志、镜像 | Key、同步、指定日重拉、概览 |
| 数据-Garmin | 上传、解析、活动列表、字段覆盖 | ZIP 上传、FIT 解析、跑步活动列表 |
| 数据-手动 | 手动录入运动 | 预留/隐藏 |
| 目标 | 当前目标、AI 澄清、历史版本 | 手动目标配置 + 版本 |
| 分析 | 新分析、重新分析、浅追问 | 新分析 + 重新分析 |
| 报告 | 当前报告、历史、对比、外部上下文 | 当前报告 + 历史列表 |
| 设置 | 档案、凭证、安全、API | 档案、训记 Key、LLM Key、登录 |
| 管理 | 用户、邀请码、映射、日志 | 预置 owner,无 UI 或极简 |

## 7. 用户路径

### 7.1 首次使用路径

```text
打开平台
→ 登录预置账号
→ 设置用户档案
→ 填训记 Key
→ 填 LLM Key
→ 同步训记
→ 上传 Garmin ZIP
→ 设置当前目标配置
→ 生成 AI 分析
→ 查看报告
```

### 7.2 日常使用路径

```text
登录
→ 同步训记
→ 上传新 Garmin ZIP
→ 如目标有变化则更新目标
→ 生成新分析
→ 查看报告和建议
```

### 7.3 重新分析路径

```text
发现报告误判
→ 补休整标注 / 更新目标 / 补数据
→ 重新分析
→ 新报告保存
→ 旧报告保留
```
