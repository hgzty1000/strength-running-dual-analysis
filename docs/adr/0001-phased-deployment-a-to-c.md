# 0001 — 分阶段部署(方案 A → 决策门 → 方案 C)+ 环境无关设计

- 状态:已接受
- 日期:2026-07-07

## 背景

双修运动平台部署在一台**阿里云轻量应用服务器**上(华东节点,大陆境内,2vCPU / 2GB / 40GB,公网 IP `106.14.241.47`,系统 Ubuntu 24.04)。

关键约束:

- **大陆节点 + 域名访问 = 必须 ICP 备案。** 备案绑定的是服务器物理位置(大陆),与域名注册地无关。未备案域名会被阿里云拦截 80/443。
- 备案是高成本动作:实名认证 + 拍照核验 + 7~20 天等待,注销时还要反向操作。
- 平台当前只给**本人 + 家人**(少数几人、分散、手机访问)使用,且尚处 **demo 验证阶段**——是否值得长期投入未知。
- 服务器仅 2GB 内存,技术栈受限。

矛盾:若一开始就备案,则为"可能被砍掉的 demo"预付时间与成本,效率低;若永久不备案(纯 IP+端口),则长期承受明文传输、地址不友好、手机端体验差的代价。

## 决策

采用**分阶段部署**,并以一条工程约束保证阶段间低成本切换:

1. **阶段一(开发/调试)——方案 A**:`http://106.14.241.47:<端口>` 直连,不备案、零成本,专注做出 demo。
2. **决策门**:demo 经本人+家人试用后,再决定是否投入上线。高成本动作(买域名、备案)一律推迟到此门之后。
3. **阶段二(仅当通过决策门)——方案 C**:阿里云购 `.com` 域名 + ICP 备案 + Let's Encrypt 正规 HTTPS。
4. **环境无关设计(贯穿始终)**:API 地址、协议、Cookie 的 Secure 策略、CORS/来源白名单等环境相关项一律走配置/环境变量,代码不写死。使 A→C 切换只需"改配置 + 装证书",不返工。

技术栈天花板(受 2GB 内存约束):轻量单体 + SQLite,demo 期不上 Docker、不上独立数据库。

## 权衡的备选方案

- **一开始就走方案 C(先备案)**:上线体验最好,但为未验证的 demo 预付两三周 + 核验成本,与"先验证再投入"的意图冲突。否决。
- **永久方案 A(纯 IP,永不备案)**:零流程,但明文传输、手机端体验差的代价"每天都在付",不适合长期承载登录 + 个人训练数据。否决为终态,仅作 demo 期手段。
- **海外/香港节点免备案**:可行但需另购服务器,且放弃现有华东机器;跨境线路有波动。当前不采用。

## 后果

- 正面:demo 阶段零流程零成本、迭代快;上线路径不被堵死;A→C 切换代价被"环境无关设计"压到最低。
- 负面:开发期为保证可切换,需从一开始就遵守"不写死环境项"的纪律,略增前期约束。
- **遗留风险(须跟踪)**:服务器刚经历一次入侵并已整机重装。当前仍为 root + 密码登录、22 端口对公网开放,且旧密码曾在沟通中出现、视为已泄露。**在 demo 放入真实数据 / 交付家人使用前,必须完成安全加固**(更换强密码,建议改 SSH 密钥登录并关闭密码登录)。此项由平台所有者手动完成。
- **凭证安全风险(须跟踪)**:多用户后,平台数据库会存储**每个用户的训记 API Key**(等同其训记账号通行证,API 支持读写)。这抬高了方案 A 明文信道的赌注——不再只是自己的数据,而是他人托付的凭证。约束:① 训记 Key 一律**加密存储**(密钥走环境变量,不进代码/不进库,符合「环境无关设计」);② **"让朋友在自己设备上填入训记 Key"这一动作推迟到方案 C(HTTPS)之后**,demo 期仅平台所有者本人在可信网络下使用,朋友注册/填 Key 的开关关闭。详见 [0003](./0003-multi-user-from-day-one.md)。
