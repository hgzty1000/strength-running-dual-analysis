# 0003 — 从第一天起按多用户建模(每用户绑定各自的训记凭证)

- 状态:已接受
- 日期:2026-07-07

## 背景

平台不止所有者本人使用,朋友也会用,且**朋友同样是训记 App 用户**。这带来两个事实:

1. 每个用户的训练数据来自**各自的训记账号**,由**各自的训记 API Key** 拉取。那个 Key(`xjllm_...`)等同其训记账号通行证——能读、能写(API 支持写回)其全部训练数据。
2. demo 阶段实际只有所有者本人使用;朋友接入被推迟到方案 C(HTTPS)之后(见「凭证安全」)。

矛盾:demo 期只有一个人用,若图省事按单用户实现(数据不带 `user_id`、Key 作全局配置),等朋友接入时要给所有历史数据补 `user_id`、重构所有查询与同步逻辑——伤筋动骨的返工。

## 决策

**把"多用户"当作数据模型地基,从第一天就建;把"是否开放朋友接入"当作一个运营开关,与建模解耦。**

1. **数据按 `user_id` 隔离**:所有运动记录(训记镜像、Garmin 活动、未来预留的手动记录)、训记凭证均归属某个 `user_id`。demo 期表里只有一行用户,但结构已就位。
2. **训记 Key 是每用户凭证,非全局配置**:用户系统里每个人各填各的 Key,平台用各自的 Key 拉各自的数据。数据隔离由此**天然成立**——A 的 Key 物理上拉不到 B 的训记数据,无需自写复杂权限逻辑。
3. **训记 Key 加密存储**:密钥走环境变量(符合「环境无关设计」),不进代码、不进库明文。
4. **朋友接入开关 demo 期关闭**:让他人在自己设备上填入训记 Key 的动作,推迟到方案 C(HTTPS)之后。demo 期仅所有者本人在可信网络下使用。

## 权衡的备选方案

- **先做单用户,朋友接入时再改多用户**:demo 期代码更少,但用户隔离是数据模型地基,事后补 `user_id` 要重构存储与全部查询,返工代价远大于现在加一个字段。否决。与「环境无关设计」同理——便宜时就把贵的坑填掉。
- **训记 Key 作全局配置(一个 Key 服务所有人)**:最省事,但违背"各拉各的数据"这一事实,且无法隔离多人数据。否决。

## 后果

- 正面:用户隔离在 demo 期几乎零成本落地(一个 `user_id` 字段 + 每用户一条加密 Key);朋友接入时只需打开开关 + 让其填 Key,无数据迁移。
- 负面:demo 期为一个用户维护了多用户结构,略增前期建模量(可接受)。
- **遗留风险(须跟踪)**:多用户后数据库存有他人训记凭证,抬高了方案 A 明文信道的赌注。故朋友填 Key 必须等 HTTPS。与 [0001](./0001-phased-deployment-a-to-c.md) 的服务器安全加固共同构成"交付他人使用前必须完成"的前置条件。
