# 0004 — 平台对外 Open API:早预留、只读存量、按 user 解耦

- 状态:已接受
- 日期:2026-07-08

## 背景

平台除了**消费**训记/佳明的数据,平台所有者还希望它**自身对外提供一套 Open API**,让外部 AI agent 能像调训记那样调本平台,从而在平台之外组合出更丰富的功能。这使平台获得**双重身份**:既是上游数据源(训记)的消费者,又是别的 agent 的上游数据源。

关键约束与事实:

- 平台已定的架构基础:多用户、数据按 `user_id` 隔离(ADR 0003)、一条数据一个主人(ADR 0002)、已有两套加密凭证(训记 Key / LLM Key)。对外 API 必须**长在这套体系上**,不能另起炉灶。
- 所有者的初步洞察:**对外 API 要提供的,本质就是"喂给 LLM 的那份结构化上下文",只是改从 API 拿走。** 二者要的是同一份数据。
- 费用敏感:LLM 调用按各用户 Key 计费,费用可控是所有者反复守的一条线。
- demo 期资源有限(2GB 服务器、精力有限),不宜为一个"未来功能"预付过多实现成本。

矛盾:若 demo 期就把对外 API 全建出来(端点、路由、Key 管理、文档),是为未验证需求预付真成本;若完全不管,则日后加 API 可能要把"上下文组装""权限判断"从已有代码里抠出来重写,伤筋动骨。

## 决策

**早预留、但不早实现;通过两层解耦把"日后加 API"的代价压到接近于零。**

1. **鉴权模型与训记同构**:每用户可签发一个"平台 API Key",绑定 `user_id`,agent 持之只能读该用户自己的数据;每用户一份、加密存储、可单独吊销(签发方向类似 LLM Key)。

2. **功能边界:只读"存量",不碰"增量"**。
   - 暴露:原始训练数据明细、加工上下文(聚合+肌群)、**当前/历史目标配置**、动作→肌群映射表、历史分析报告、休整标注、元数据。
   - 排除:写入(包括外部 agent 写回训练计划、目标、标注、报告等,这根线不碰);**让 agent 触发分析**(会消耗 LLM 额度、生成新报告,构成**连锁调用 → 费用不可控**,明确不做)。
   - 原则:读存量零边际成本;凡产生增量/花额度/改状态的一律排除。

3. **"预留"= demo 期摆正两层解耦,不建实际端点**:
   - **上下文生产独立成层**:"库原始数据 → 结构化上下文"的组装逻辑独立出来,AI 分析与未来 API 共用同一生产者,不塞进"分析按钮"的代码路径。外部 agent 主要消费的是加工后的训练上下文 + 当前目标配置/目标版本 + 必要历史分析结论,而不是数据库全量 dump。
   - **数据访问按 `user_id` 解耦,不绑死网页 session**:底层只认当前 `user_id`;其来源(网页 cookie / API Key)由上层解析。网页与未来 API 共用同一套按 user 取数逻辑。
   - 这两层都是为 AI 分析/多用户**本就要写**的代码,只是放对位置,额外成本极低。HTTP 路由与 Key 管理界面日后是"做加法",不返工。

## 权衡的备选方案

- **demo 期就完整实现对外 API**:一步到位,但为未验证需求预付端点/鉴权/文档成本,与"先验证再投入"的一贯精神(ADR 0001 决策门)冲突。否决。
- **完全不考虑,日后再说**:demo 期最省,但极可能把上下文组装、权限判断写死在分析功能与网页 session 里,日后加 API 需重构核心数据访问路径,伤筋动骨。否决。
- **只做"上下文层"解耦、不管鉴权解耦**:漏了一半——若权限判断绑死浏览器 cookie,日后 API(用 Key 而非 cookie)仍要重写权限层。故必须两层一起摆正。
- **对外 API 支持写入 / 触发分析**:功能更强,但写入冲击数据主权、触发分析导致费用不可控。外部 agent 写回这根线明确不碰;触发分析同样否决,只读先行。

## 后果

- 正面:demo 期近乎零额外成本即为对外 API 预留好位置;日后加 API 只需"做加法";鉴权天然复用多用户隔离,不引入新信任模型;只读存量边界把费用与数据主权风险挡在门外。外部 agent 可读取训练上下文、目标配置和历史分析做更深入的多轮方案讨论,但其输出不回写平台,平台也不替外部 agent 触发新的 LLM 分析。
- 负面:demo 期需守两条解耦纪律(上下文层、数据访问层),略增前期结构约束——与「环境无关设计」「多用户从第一天建模」同源的"便宜时先填坑"。
- **遗留待议**:若未来出现明确场景,可单独讨论更复杂的费用/额度模型;但外部 agent 写回平台不在当前路线内,本 ADR 将其明确排除。
