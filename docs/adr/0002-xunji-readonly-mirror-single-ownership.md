# 0002 — 训记为只读镜像 + 一条数据一个主人 + 不为历史改动买单的同步策略

- 状态:已接受
- 日期:2026-07-07

## 背景

本平台定位为运动数据的**分析层**,而非录入工具。力量训练数据的主要来源是**训记 App**——已实测其 Open API(`https://trains.xunjiapp.cn`)可拉取完整力量数据:动作中文名、每组重量/单位/次数/RPE/组间休息、`done` 标记、训练热量、精确起止时间戳。手动录入能力仅预留,demo 阶段不开发;若未来开发,也只用于外部数据源没有覆盖的运动事实。

由此产生三个必须现在就定的数据模型问题:

1. **同一条力量数据,会不会既能从训记同步、又能在本平台手动改?** 若能,就要处理双向同步与冲突合并。
2. **API 有限频**(轻量读 15s/次、完整读 30s/次、写回 45s/次),不能实时逐天现拉现算。
3. **API 不提供任何服务端变动信号**——实测训练/动作/组三层字段(`localid/datestr/title/note/start/end/started_at/ended_at/movements` 等)均无 `updated_at`、版本号或哈希。因此"某天是否被改过"服务端无从告知,只能靠"重拉那天 + 与本地比对",成本等同于直接重拉。

用户的行为事实:一个月以上的历史记录几乎不变;一个月内偶有补改(改重量、补 RPE),但每月不超过两次。

## 决策

**一、训记为只读镜像(Xunji Mirror)**
从训记同步进来的力量记录,在本平台内**只读**。主人是训记——要改回训记 App 改,下次同步以训记为准。demo 期本平台对训记**只读不写回**,避免污染用户在训记里的真实训练数据。

**二、一条数据一个主人(Single Ownership)**
任意一条运动记录**有且只有一个主人**,编辑权互斥,两类数据永不重叠:
- 训记同步来的 → 训记拥有,本平台只读镜像。
- 手动录入的(未来预留,demo 不开发) → 本平台拥有,录的是外部数据源没有覆盖的运动事实,而非覆盖训记/佳明数据。

分析时把所有来源**合并呈现**,但**编辑权按来源隔离**。直接后果:本平台**不需要实现双向同步与冲突合并**。

**三、同步策略:不为罕见的历史改动买单**
把同步拆成两个互不相同的操作:
- **增量同步(常规按钮)**:只拉"上次同步日当天 → 今天"这些天。因当天可能多次训练/多次同步,**最近一天永远重拉一次**,避免漏掉当天后续训练。这是高频操作,快。
- **指定日重拉(补丁入口)**:用户自己知道改了哪天,手动输入日期,平台只重拉那一天覆盖本地镜像。为每月一两次的罕见历史改动兜底,代价只在真用时才付。

**四、本地缓存/同步层**
因限频,训记数据同步进**本地存储**(demo 期 SQLite),分析在本地数据上跑,不实时现拉。按 `datestr` 缓存,同一天不重复请求。

## 权衡的备选方案

- **设计乙:本平台作统一编辑层,训记只作初始导入。** 体验上单一编辑入口更统一,但要实现双向同步 + 冲突解决(「训记说 60kg、本平台被改成 65kg,信谁?」),且写回会污染训记真实记录。demo 期是巨大的坑。否决。
- **滚动 30 天重扫窗口:** 每次同步自动重拉最近 30 天并逐天比对哈希,以自动捕捉一个月内的补改。但空白天也占请求,最坏 30 次 ×15s ≈ 7-8 分钟;且为"每月不超两次"的罕见事件让每次常规同步都买单,代价不成比例。否决,改由"指定日重拉"手动兜底。
- **依赖服务端变动信号做精准增量:** 最理想,但实测 API 不提供任何 `updated_at`/版本号/哈希,方案不成立。

## 后果

- 正面:从根上消除同步冲突;常规同步是纯增量,飞快;历史改动的成本只在真发生时才付。
- 负面:用户若忘记自己改过某天的历史记录、又没用"指定日重拉",本平台镜像会与训记短暂不一致,直到下次手动重拉或全量重扫。这是刻意接受的取舍。
- 遗留:手动录入能力仅预留,demo 阶段不开发。未来若开发,其与外部来源镜像"永不重叠"依赖产品边界与必要校验共同保证。
