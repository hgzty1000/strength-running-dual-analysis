# 讨论备份 — 双修运动平台可行性讨论

- 日期:2026-07-08
- 用途:**聊天记录的持久化备份**,防止对话丢失后需重头再想。
- 说明:本文件是**讨论过程**的存档,不是权威定义。已锁定的术语以 [CONTEXT.md](../../CONTEXT.md) 为准;已定的架构决策以 [docs/adr/](../adr/) 为准。本文件额外保留**弯路、被否方案、和尚未拍板的悬案**——这些不进 CONTEXT.md,但对理解"为什么这么定"有价值。

## 一、这场讨论的主线

用户初始需求:做一套软件,**全面管理自己的运动(撸铁 + 马拉松双线)和饮食**,让两条线都能进步。市面软件因"撸铁+马拉松"非常规组合而无法指导。

讨论按**可行性优先**推进,顺序:①数据怎么获取 → ②AI 分析价值验证 → ③(未开始)饮食线。

产品定位在讨论早期收敛为:**"记录 → AI 分析合理性 → 给建议"的反应式分析工具**(非被动记录本,也非主动出计划的教练)。

## 二、已锁定的结论(详见 CONTEXT.md)

### 数据获取(可行性已闭环)

| 数据线 | 结论 | 状态 |
|---|---|---|
| 撸铁 | 训记 App Open API(已用真实 Key 实测,字段齐全) | ✅ 通 |
| 跑步 | 佳明手动导出 FIT 文件(`.zip` 内含 `.fit`)→ 网页端上传解析;解析层按通用 Garmin 活动入口设计 | ✅ 方案定 |
| 饮食 | 饮食**记录**砍掉(暂用训记功能);饮食**建议**以「营养素建议」保留(见第八节) | ✅ 已决定 |
| 手动记录 | 能力预留,demo 阶段先不开发(模板尚未想清,避免做成万能日记本) | ⏸️ 预留 |

关键实测事实:
- 训记返回力量数据完整:动作中文名、每组重量/次数/RPE/组间休息/`done` 标记/训练热量/精确时间戳。
- 训记有**动作目录接口**(259 个动作,带肌群 `type` 分类),但**相当多动作 `type` 为空**(验证中 72 组落入"未分类")。
- 训记 API **无任何服务端变动信号**(无 `updated_at`/版本号/哈希),故无法低成本探知历史被改。
- 训记限频:完整读 30s/次、轻量读 15s/次、写回 45s/次。

### 数据模型 & 架构

- **一条数据一个主人**(Single Ownership):训记镜像只读(主人是训记)、Garmin 文件上传活动归 Garmin 文件来源;手动记录仅预留,demo 不开发。任意运动事实只能有一个主人 → 从根上免除同步冲突。(ADR 0002)
- **同步策略**:手动触发;增量同步(拉"上次同步日当天→今天",最近一天永远重拉);指定日重拉作历史改动补丁;不为罕见历史改动做滚动重扫。
- **多用户从第一天建模**:数据按 `user_id` 隔离,每用户绑各自训记 Key;"开放朋友接入"是运营开关(demo 期关,HTTPS 后开)。(ADR 0003)
- **两套凭证信任模型**:训记 Key(用户自己提供、等同账号通行证)vs LLM Key(平台所有者签发下发、用于用量归属+可单独吊销)。均加密存储。

### AI 分析(云端)

- 走**外部云端大模型 API**(硬件决定:服务器 2GB 跑不动本地模型);统一模型商、平台级地址;用户主动触发、只发必要片段。

### 分析规则(由 2026-07-08 真实数据验证逼出)

1. **训练中断上下文(休整标注)**:见下方专节,已细化为"按需纠偏工具"。
2. **容量优先,弃用训记时间戳**:训练量以容量(组×次×重量)算;时间戳因用户忘开/忘关不可靠,弃用。
3. **动作肌群标注:AI 补全 + 固化复用**:训记没给肌群的动作,AI 判定一次并固化进"动作→肌群"映射表,以后复用,不每次现场重判。
4. **以个人基线为参照,而非绝对标准**:拿用户和自己的历史节奏比,不套教科书死标准(因其为非常规训练)。
5. **分析可迭代 + 分析对话不产生数据**:补上下文→重新生成;支持当前报告内的浅追问,但分析层只读数据,绝不从交互中新增/改运动数据。

## 三、AI 分析价值验证的实况(2026-07-08)

用本人 14 天(10 个训练日)真实训记数据跑了一次样例分析。**结论:价值成立,但三条样例结论错了两条——而这两个错恰恰逼出了核心规则。**

- ❌ 误判"马拉松却不练腿" → 真相:腿本是周 1 次,因**膝手术**轮空。→ 逼出「训练中断上下文」。
- ❌ 误判"单次时长忽长忽短" → 真相:训记忘开/忘关,**时间戳不可靠**。→ 逼出「容量优先,弃用时间戳」。
- ✅ 正确命中"推拉失衡约 5:3(推 101 组 vs 拉 61 组)" → 证明数据干净时,AI 能说出通用软件不会说的、具体的话。

**核心发现:AI 分析可行,但其正确性高度依赖"喂给它的上下文够不够"——尤其是伤病/中断这类"数据里看不出、但决定判断对错"的背景。**

## 四、休整期(训练中断上下文)—— 已钻透

经多轮对齐,最终形态:

- **本质**:以**自由备注**为主承载"这段怎么回事",预设几个 **tag** 作快速录入(如"因病停跑""因病停力量""出差停跑")。tag 方案甲:**预设打底 + 可自由加**。
- **tag 按训练线免责**:"停跑"只对跑步线的空缺免责,不影响评判同期力量训练(用户这次正是"停腿/停跑但上肢照练")。
- **按需纠偏,而非事前维护**:用户**平时什么都不用标**。仅当 ① AI 就某空缺发问,或 ② AI 建议明显失准时,用户才回头补一个休整标注,**触发重新生成判断**。
- **默认不追究单日空缺**:练一休一的休息日是训练的一部分。AI 只在**偏离个人基线**(如平时周练腿1次、突然连续数周没练)时"存疑地问",而非"武断地判"。备注用来提前消解这种问询。

## 五、✅ 已闭合:AI 分析的"范围"与交互形态

**曾是本场最关键的悬案。用户原话:"这个范围很难定的合适,如果太死板,用处不大,太灵活,又很复杂。"** 经数轮对齐,已拍板。最终结论见 CONTEXT.md「分析报告」「分析可迭代」。

### 问题
AI 一次分析,输入范围该是多大?单条 / 单日 / 滚动周 / 月?交互是按钮还是对话?结果怎么存?

### 关键转折(对齐过程)
1. **范围两难的解法**:不让人事先框死范围,而是**默认喂宽窗口(约 6 周)+ 让 AI 在生成报告时内部自主决定聚焦尺度**(该讲趋势讲趋势、该点某次点某次)。能交给 AI 判断的,不让用户/开发者硬框。
2. **"会不会变成聊天窗口"的顾虑,被一条更聪明的规则化解**:用户提出**"对话数据不影响本来的数据,运动数据以数据库为准"**——即防污染靠**架构分层(分析层只读、对话产出永不回写数据层)**,而非靠"砍掉输入框"。于是自由提问的口子可以安全放开。
3. **"上下文冲突"两难的拆解**:把"上下文"拆成两层——**数据上下文**(每轮从库实时重取,永不冻结进历史 → 无"旧数据 vs 新数据"冲突)与**对话上下文**(保留话题线索,让追问不愚蠢)。冲突来自缓存了旧数据,不来自记住了话题。
4. **多轮对话程度**:定为**第 1 档「浅追问」**——只记当前这次分析的话题线索,可追问"那胸呢";点**"新分析"按钮**清空重来。不做会话级/跨会话记忆(过重、易发散)。
5. **结果形态**:定为**形态甲——每次分析产出一份可存档的结构化报告**(而非"对话即分析")。理由是用户强调**复盘很重要**,而对话记录无法结构化保存、无法做趋势比较与个人基线。

### 最终拍板(已写入 CONTEXT.md)
- **分析产物 = 结构化报告(主)+ 浅追问(辅,不改报告定论)**。
- 报告**双层存储**:结构化字段层(供比较/趋势/基线)+ 叙述层(供阅读)。
- **默认宽窗口(~6 周)+ AI 内部自主定聚焦尺度**;可带自由文本指定关注点;可选档位/自定义区间做历史回溯。
- **浅追问(第1档)+ "新分析"按钮重置**。
- 防污染铁律:**分析层只读、对话产出永不回写;运动数据一律以数据库为准**(强化「分析对话不产生数据」)。
- 防冲突铁律:**数据每轮从库实时重取,绝不冻结进对话历史**。

## 六、平台对外 API(预留)—— 已定边界(ADR 0004)

用户声明:平台需**预留一套对外 Open API,让其他 AI agent 能像调训记那样调本平台**,以组合更丰富的功能。平台由此获得**双重身份**:既是训记的下游消费者,又是外部 agent 的上游数据源。

关键对齐结论:

- **鉴权**:沿用训记模型——**每用户签发一个平台 API Key,按 `user_id` 隔离、加密存储、可单独吊销**。天然复用「多用户隔离」(ADR 0003)。
- **只读**:demo 期**只读**。写入"确实复杂"(agent 写入的数据算谁的、如何不破坏「一条数据一个主人」),用户明确要求押后专门讨论。
- **"预留"的确切含义 = 两条解耦约束**(demo 期不建实际 API 端点 / Key 管理界面,但结构要留好):
  1. **上下文生产独立成层**:AI 分析与未来 API **共用**同一个"原始数据 → 结构化上下文"的生产层。源于用户洞察——"API 拿的就是喂 LLM 的那份上下文"。因这段逻辑分析本来就要写,独立放置**零额外成本**。
  2. **数据访问按 `user_id` 解耦,不绑死网页 session**:底层只认"当前 user_id 是谁",至于来自网页 cookie 还是 API Key 由上层解析 → 网页与未来 API 共用同一套按 user 取数逻辑。
- **功能边界(读存量,不碰增量)**:
  - **纳入(第一、二档)**:原始训练数据、加工上下文、**当前/历史目标配置**、动作→肌群映射表、历史分析报告、休整标注、元数据(数据概览/日期覆盖/上次同步)。
  - **排除(第三档及以上)**:**"触发分析"被明确否决**——会花 LLM 额度、可能连锁调用,**费用不可控**(用户原话)。**外部 agent 写回同样明确不做,这根线不碰**(包括写回训练计划、目标、标注、报告等)。
- **典型用途补充**:平台内置建议只做方向性/决策级调整;更细的方案推演、多轮教练式讨论,可由外部 agent 通过只读 API 读取训练上下文 + 目标配置 + 历史分析后完成。外部 agent 输出不自动回写平台,也不能触发平台 LLM 费用。
- **原则**:对外 API 只读暴露"平台已拥有/已算好的存量";凡"产生增量 / 花额度 / 改状态"的一律排除。

> 详见 [CONTEXT.md](../../CONTEXT.md)「平台对外 API」与 [ADR 0004](../adr/0004-outbound-api-reserved-readonly.md)。

## 七、AI 辅助目标澄清—— 已定为独立模块(2026-07-08)

在回到第一性原理讨论后,平台核心进一步收敛为:**基于个人数据和当前目标配置,评估力量 + 跑步双线训练的平衡、冲突与过量风险,并给出调整方向**。

关键痛点由用户明确表达:
- 力量训练和跑步训练目前是割裂安排的。
- 用户不知道两者如何平衡,主要靠体感估计。
- 因体感滞后,经常出现训练过量。
- 力量目标没有明确数字,更像"保持体型、保持当年的肌肉量";跑步目标则常有配速、全马成绩、比赛日等明确数字与日期。

因此,"目标"不能用一个统一模板套住。最终对齐为:

### 目标配置,不是固定模板

平台需要的是「当前目标配置」,作为 AI 判断"合理/过量/谁让步"时的判断尺。结构为:
- **主目标**:双线平衡、跑步比赛/成绩优先、力量/体型优先、恢复优先、减脂/体重控制优先、自定义。
- **目标详情**:随主目标变化。跑步可有距离、比赛日、目标成绩/配速、备赛阶段;力量可为维持体型/肌肉量、关注部位、是否接受体重变化等。
- **底线约束**:表达"让步但不放弃"。例如跑步比赛季中,腿训可让步为维护量,但体型相关的胸/背/肩/手臂容量不能长期跌出个人基线。
- **冲突取舍**:当跑步、力量、恢复、体重目标冲突时,明确谁优先、谁让步、让步到什么程度。
- **不确定项**:允许目标先粗后细,把尚未明确的比赛日、目标成绩、可接受的力量下降幅度等标为待补。

### AI 目标澄清是独立模块

用户明确同意:需要增加**AI 辅助目标澄清**,并且这是一个**单独模块**,不和之前的 AI 分析混在一起。

边界已定:
- **目标澄清 ≠ 分析追问**:
  - 分析追问:解释当前报告,不改目标。
  - 目标澄清:通过一问一答生成/修改目标配置。
- **AI 生成草案,用户确认后才生效**:不能因为用户随口聊天就自动改目标;必须展示目标摘要,由用户确认/修改后保存。
- **目标配置版本化**:每次确认生成一个目标版本,记录生效日期、替换/结束日期、摘要、详情、生成方式(AI 访谈/手动编辑)、用户确认时间。历史报告引用当时的目标版本。
- **不修改运动数据**:AI 目标澄清只写目标配置,不新增/修改训练记录。demo 阶段运动数据只来自训记镜像与佳明文件上传;手动记录仅预留,未来若开发也只记录外部数据源没有覆盖的运动事实。
- **可同时建议休整标注,但分开保存**:如膝部手术可同时产生一条休整标注解释过去缺训,以及一个恢复优先目标配置指导未来判断,但二者职责不同、分别确认。

一句话锁定:**目标可以由 AI 对话生成,但必须显式确认、版本化保存、与运动数据隔离。**

> 详见 [CONTEXT.md](../../CONTEXT.md)「当前目标配置」「AI 目标澄清」。

## 八、饮食决定:砍记录、留建议、加档案(2026-07-08)

用户对饮食这块想清楚了三件事,必须拆开——不能让"砍饮食"把不该砍的也带走:

1. **砍掉「饮食管理/记录」**:记录"每顿吃了什么"因无现成源、手动录入反人性、用户尚未想清 → 移出范围。饮食记录暂用**训记自带饮食功能**顶替。
2. **保留「营养素建议」(实为 AI 建议的扩展,非饮食记录残留)**:根据当天运动量给营养素目标(碳水/蛋白/脂肪 g、总热量)。**关键区别**:饮食记录难在"输入",营养素建议是"输出"——它**只消费运动数据 + 用户档案,零饮食录入**,因此绕开了饮食那块所有难点。
3. **新增「用户档案」**:为营养素建议提供计算依据。

- **用户档案最终字段(经"从必需到可选"分层敲定,并在目标配置独立后修正)**:
  - **必需(BMR 公式 Mifflin-St Jeor 要用)**:身高、体重、**年龄、性别**(后两者是补充的——缺了 BMR 算不准,且都是"填一次基本不变"的静态项)。
  - **不再在用户档案里存目标**:原先讨论过"目标:增肌/减脂/维持"可选字段;在「当前目标配置」独立成模块后,该字段从用户档案删除,避免"档案目标"与"当前目标"两套判断尺冲突。训练/营养目标统一由当前目标配置表达。
  - **方案甲:只存当前值,不追踪体重趋势**(体重变化不快,手动修正即可;做时间序列会把"填一次的资料"升级成"又一条要维护的数据线",与砍饮食精神矛盾)。
  - **第三层精修(体脂率/睡眠/静息心率等)明确不做**:用户判断"这些数据很难拿到正确的,反而无用"——与「弃用训记时间戳」同一原则(不可靠的输入不如不用)。

**关键洞察:平台不需要传统营养 App 的"活动系数"(让用户手选久坐/轻度/中度)**——因为训练消耗可由训记/佳明**实测数据**估算,比手选活动等级准。这是本平台强于通用营养 App 之处。

营养素建议的边界也已补充:
- 宏量营养素数字应尽量基于可解释公式/规则(用户档案 + 训练消耗 + 当前目标配置),AI 负责解释和微调,不凭空随口报数。
- 不生成饮食菜单,不记录实际饮食,不精确到每顿,不替代医学营养建议。

> 详见 [CONTEXT.md](../../CONTEXT.md)「用户档案」「营养素建议」。

## 九、跑步数据分析边界—— 不做小 Garmin,服务双线总控(2026-07-08)

在佳明数据源确定为 FIT 文件上传后,进一步讨论了跑步数据进入平台后到底分析什么。

核心结论:
- 平台**不复刻 Garmin**,不做一个“小 Garmin”。Garmin 已经负责专业跑步指标展示、训练状态、地图等。
- 本平台使用跑步数据的目的,是把跑步活动转化为双线负荷的一部分,服务于:**过量判断、下肢冲突、恢复压力、营养需求和目标取舍**。
- 因此跑步分析服从「双线总控」,而不是孤立做专业跑步教练。
- **FIT 入口按通用 Garmin 活动数据源设计,不写死为跑步专用入口**。后续可能有骑行、游泳、越野跑、徒步等 FIT;通用解析层先解析活动事实,再按 `sport/sub_sport` 标准化为 activity family / variant,最后交给运动类型专属解释器。demo 阶段只对 running family 做完整分析;非跑步 FIT 只做通用解析、查重、存档、类型标记和元数据展示,暂不进入双线负荷矩阵或专项建议。

### FIT demo 期目标解析字段

demo 期目标解析并存储:
- 活动时间、活动类型、距离、时长/移动时长、配速/速度、消耗热量。
- 心率:平均心率、最大心率、心率区间/逐点可算则用。
- 分段/lap 或配速波动:用于识别跑步类型。
- 海拔爬升/下降:用于解释坡度带来的负荷差异。
- **步频、功率、气温**:用户明确要求纳入核心字段。

字段策略:
- 不同 FIT 不保证都有全部字段。
- 平台应有则解析/存储/纳入分析;没有则标记缺失,并在报告中声明影响。
- 步频用于辅助判断疲劳和跑步状态,但不做高级跑姿诊断。
- 功率用于更准确解释真实强度,但不做功率训练处方。
- 气温用于解释心率/配速/疲劳变化,但不做热应激医学模型。

### 跑步活动归类

用户确认需要跑步类型标签。最终分为两层:

1. **训练类型**:轻松跑、长距离跑、节奏跑、间歇/速度课、恢复跑、比赛/测试、未知/混合。
2. **跑步场景**:户外路跑、体育场/操场跑、跑步机、未知。

跑步机需要单独作为场景标签,因为它不同于户外跑:
- 没有 GPS 轨迹。
- 距离/配速可能来自手表估算或跑步机,置信度不同。
- 海拔通常无意义。
- 气温可能不代表真实室内环境。
- 但它仍然计入有氧负荷、下肢负荷、全身恢复负荷和营养需求。

体育场/操场跑也需要单独作为场景标签:
- 有 GPS,但路径范围小,通常是重复绕圈。
- 海拔通常无意义。
- 常用于间歇、节奏、测试等强度训练。
- 正常计入跑步负荷和下肢负荷,但解释配速波动、GPS 轨迹和爬升时应按场地跑处理。

### FIT 实测验证

2026-07-08 已用工作区 3 个真实 Garmin 导出 zip 验证 FIT 链路:

| 文件 | 场景 | 关键识别信号 | 解析结果 |
|---|---|---|---|
| `614314322.zip` | 跑步机 | `sub_sport=1`, GPS 点 0, 无海拔 | 可取得距离、elapsed/timer time、配速、心率、步频、功率、消耗、lap |
| `605823485.zip` | 户外路跑 | `sub_sport=0`, GPS 点约 6085, 有海拔起伏 | 可取得距离、elapsed/timer time、配速、心率、步频、功率、海拔、消耗、lap |
| `607249844.zip` | 体育场/操场跑 | `sub_sport=4`, GPS 点约 2193, 经纬度范围小, 海拔基本不变 | 可取得距离、elapsed/timer time、配速、心率、步频、功率、消耗、lap |

补充结论:
- Garmin zip 结构为一个 zip 内含一个 `*_ACTIVITY.fit`,上传 zip → 解压 → 解析 FIT 可行。
- `elapsed time` 与 `timer time` 会明显不同,二者都应保存;训练负荷/配速优先参考 timer time,elapsed time 用作活动跨度参考。
- 3 个样本都没有气温字段,故气温作为目标字段保留,但必须允许缺失并在分析中说明。
- 跑步机/户外路跑/体育场跑可通过 `sub_sport` + GPS/海拔特征做 demo 期场景识别。

### demo 阶段明确不做

- 不复刻 Garmin 训练状态。
- 不自研 VO2max 估算。
- 不做跑姿技术诊断。
- 不做地图轨迹展示。
- 不做完整跑步教练计划。
- 不做功率训练区间处方。
- 不做热应激医学模型。
