# CONTEXT — 力跑双训分析系统

本文件记录本项目领域内有明确含义的术语。只收录对"业务/领域"有意义的词,不记实现细节。

## 术语表

### 双修
指**有氧训练 + 力量训练**两类训练的组合,而非字面或其他引申含义。
典型场景:用户同时进行健身(力量)与马拉松训练(有氧)。

> 澄清历史:项目名"双修"最初存在歧义(可能被理解为双人/亲密类含义)。已明确锁定为"有氧+力量"的训练组合含义。本平台内容属纯健身/训练类,无敏感内容合规风险。

### 双修运动平台
本项目本体。一个网页端、云端部署的个人训练平台,供**用户本人及家人**使用(分散、各自手机随时访问)。
定位为**分析层**而非录入工具:训练数据主要来自外部数据源(见「数据来源」),本平台负责**同步 → 缓存 → 分析 → 建议**。手动录入只作能力预留,demo 阶段先不开发(模板尚未想清,避免把入口做成万能日记本)。

### 核心能力(产品定位)
本平台的核心工作流是**"记录 → AI 分析 → 建议"**的三段式,且为**反应式(reactive)**:
1. 用户的**已发生**运动数据进入平台(训记同步 / 佳明文件;手动录入仅预留, demo 阶段不开发);
2. AI 对已录入数据做**合理性分析**;
3. 基于分析给出**建议**(训练建议 + 见「营养素建议」)。
> 明确边界:demo 阶段**不做主动出计划的教练**(不预先生成"今天该练什么"),只对已发生的事后分析并建议。主动排程(proactive coaching)如需要,推迟到方法论成文 + 决策门之后。

> **饮食管理已移出范围(2026-07-08)**:记录"每顿吃了什么"这条线因无现成数据源、手动录入反人性、且用户尚未想清而**砍掉**;饮食记录暂用**训记自带的饮食功能**顶替。注意:被砍的是**饮食记录(输入)**,而非**营养素建议(输出)**——后者只消费运动数据 + 用户档案,零饮食录入,保留在「营养素建议」。

### 当前目标配置 (Current Goal Configuration)
平台判断"训练是否合理"时使用的**判断尺**,不是一套固定目标模板。原因是本项目的两条训练线目标形态不同:跑步常有赛事/成绩/配速/比赛日这类**结果型目标**,力量训练当前更偏"保持体型、保持肌肉量"这类**维持型/底线型目标**。

当前目标配置采用**主目标 + 底线约束 + 冲突取舍 + 不确定项**的结构:
- **主目标**:当前阶段最优先的方向,如双线平衡、跑步比赛/成绩优先、力量/体型优先、恢复优先、减脂/体重控制优先、自定义。
- **目标详情**:按目标类型变化。跑步可记录距离、赛事日期、目标成绩/配速、备赛阶段;力量可记录维持体型/增肌/力量提升、关注部位、是否接受体重上升等。
- **底线约束**:表达"让步但不放弃"。例如跑步比赛季中,腿训可让步为维护量,但胸/背/肩/手臂体型相关容量不能长期掉出个人基线;也可约束关键跑课前 24-48 小时避免高容量腿训。
- **冲突取舍**:当跑步、力量、恢复、体重目标互相冲突时,平台按当前配置判断谁优先、谁让步、让步到什么程度。
- **不确定项**:目标尚未想清或字段缺失时允许先保存粗配置,并标记待补问题(如比赛日、目标成绩、可接受的力量下降幅度)。

目标配置是**低频变化**的背景,不是每日维护项;允许先粗后细,由使用过程逐步精细化。每份分析报告应引用当时生效的目标配置,避免用后来目标解释历史报告。

### AI 目标澄清 (AI Goal Clarification)
一个**独立于 AI 分析**的模块。平台通过对话帮助用户澄清当前训练意图、比赛目标、力量底线、恢复约束和冲突取舍,生成一份待确认的「当前目标配置」。

- **独立入口,不混入分析追问**:目标澄清用于生成/修改目标配置;分析追问只解释当前报告。两者逻辑上必须分开,避免用户一句闲聊被误当成目标变更。
- **与休整标注分工**:过去某段异常/缺训的事实解释存「休整标注」;当前/未来的判断尺存「当前目标配置」;如果同一事件两者都影响,则由 AI 建议拆成两个对象分别确认、分别保存。
- **AI 生成草案,用户确认后才生效**:目标访谈可以产出结构化草案,但不能自动保存。必须展示摘要,由用户确认/修改后才成为当前目标配置。
- **版本化保存**:目标配置一经确认即形成版本,包含生效日期、替换/结束日期、目标摘要、目标详情、生成方式(AI 访谈/手动编辑)、用户确认时间。历史版本不无痕覆盖。
- **不修改运动数据**:目标澄清可以生成/修改目标配置,但绝不新增或修改训练记录。demo 阶段运动数据只来自训记镜像与佳明文件上传;手动记录仅预留,未来若开发也只记录外部数据源没有覆盖的运动事实。
- **可同时建议其他对象,但分开保存**:例如膝部手术场景中,AI 可建议同时保存一条「休整标注」解释过去缺训,以及一个「恢复优先」目标配置指导后续判断;二者职责不同、分别确认。

> 与「分析对话不产生数据」的关系:目标配置不是运动数据。AI 可以帮助生成目标,但必须显式确认、版本化保存、与运动数据隔离。

#### v0.2 实现范围(demo 落地口径)

上面是完整版设想,本次 v0.2 按以下口径实现,取舍已与用户确认:

- **入口**:「目标」页新增澄清对话入口,与手动填写表单并存。
- **多轮追问**:围绕主目标倾向、跑步目标(比赛日/目标成绩)、力量底线(可接受下降幅度)、冲突取舍、恢复约束展开;追问**带不确定性、不武断**(存疑而非指责)。
- **喂近期训练概况**:澄清时把用户近期训练概况(月容量、跑量、跑步类型分布,复用 `month_calendar` 等现有聚合)作为背景喂给 AI,使追问贴合实际。纯只读消费,不产生数据。
- **草案 → 可编辑表单 → 确认**:AI 汇成结构化草案,字段对齐现有目标表(`primary_goal` / `running_goal_text` / `strength_baseline_text` / `conflict_policy_text` / `uncertainties_text` / `effective_from`);没聊清的字段进 `uncertainties_text`,不硬填。草案以**可编辑表单**呈现,用户可改任意字段再确认。确认走现有版本化落库逻辑,`created_by` 标记为 `ai_clarification`(区别于手动的 `manual`)。
- **不存对话**:对话仅在会话内进行,**只落最终确认的目标版本**,过程不建表、不落库。
- **无 LLM 兜底**:未配 LLM Key 时对话入口置灰,提示去「设置→凭证」配 Key;手动表单依旧可用。
- **休整拆分仅提示不代建**:AI 识别出"过去异常/缺训"应归休整标注时,提示"这像是休整事件,建议去休整标注记一笔"并给链接,**不自动创建**休整标注。

### AI 分析(云端) (Cloud AI Analysis)
"记录 → **AI 分析** → 建议"中的分析环节,由**外部云端大模型 API** 承担,本平台不跑本地模型。
- **硬件决定,非偏好**:服务器仅 2GB 内存(见 ADR 0001),跑不动任何本地大模型。本平台只负责「取本地缓存的训练数据 → 拼提示词 → 调云端 API → 存/展示建议」。
- **统一模型商、平台级地址**:全平台接入**同一家**大模型商(例:DeepSeek),模型商地址与模型名是**平台级配置**(走环境变量,见「环境无关设计」),对所有用户一致。
- **数据外流边界**:分析会把训练数据发往第三方模型商。故设计为**用户主动触发 + 只发必要片段**(点"分析这周训练"时才把那一段发出去),而非每条记录自动上传。把外流控制在最小、最可控。

> 以下「分析规则」由 2026-07-08 一次真实数据验证逼出——用本人 14 天训记数据跑样例分析时暴露的失真点。它们是 AI 分析要正确所必须遵守的领域约束。

### 休整标注 (Rest/Interruption Note)
一个**按需的纠偏工具**,不是需要日常维护的状态。用于向 AI 解释"某段训练空缺是有原因的"(伤病/手术/出差…),避免误判。
- **为何必须有**:验证中,AI 见"14 天只练 1 次腿"即判"马拉松却不练腿"——实为膝部手术轮空。**这种误判"看起来很对、实则完全错",比数据缺失更危险。**
- **触发式,非预防式(关键)**:用户**平时什么都不用标**。仅在两种情况下回头补标注 →(a)AI 就某空缺发问时,(b)AI 建议明显失准时。标注后**触发重新分析**(见「分析可迭代」),让 AI 带着背景重跑。**不误解就永远不用标**,标注负担压到最低。
  - 这解决了"没练又没标怎么办":绝大多数空缺在个人常态内(见「个人基线为参照」),AI 默认**不追究**;只有偏离基线时才**存疑发问**(而非武断指责);休整标注用于**消解**这个发问。
- **形态:自由备注为主 + 预设 tag 加速**。tag 用于快速录入高频场景,预设打底(如"因病停跑""因病停力量""出差停跑")**且可自由新增**(方案甲,与「动作肌群标注」同为"用起来才长出来")。
- **tag 按训练线免责,不波及另一条线**:tag 天然是「部位/线 × 原因」组合。如"因病停跑"仅对**跑步线**的空缺免责,**不影响** AI 评判同期的**力量**训练(用户本次即"停跑/停腿但上肢照练")。
- **休整标注是"元信息",非训练数据**:它是分析时补充的上下文,不构成一条运动记录,不违反「分析对话不产生数据」。

### 容量优先,弃用训记时间戳 (Volume-first, Distrust Timestamps)
训练量一律以**容量(组 × 次 × 重量,tonnage)**衡量,**不采信训记的起止时间戳**做训练时长分析。
- **为何弃用时间戳**:用户常忘开(补录 → 时长偏短)、忘关(时长偏长),`start`/`end` 不可靠。验证中"单次时长忽长忽短"的结论即因此作废。
- **推论**:仅数"组数"也失真(会把"深蹲 4 组 100kg"与"提踵 4 组空杆"当等价)。容量计算必须纳入每组**重量**;强度参考 **RPE**(训记 `sets[].rpe`,当有值时)。

### 动作肌群标注:AI 补全 + 固化复用 (Muscle-group Tagging)
给「训记镜像」补一个"肌群标注层":训记动作目录给了肌群(`type`)的**直接用**;**没给的(验证中 72 组落入"未分类")由 AI 判定补全**。
- **关键约束:AI 补一次,固化进平台的"动作 → 肌群"映射表,以后复用,不每次现场重判。** 理由:①每动作只花一次 AI 判定成本(按 Key 计费);②**同一动作分类永远一致**,历史训练量统计不会忽上忽下;③快。
- 映射表**由 AI 自动长出**,无需用户手动维护 259+ 动作;用户可选地手动订正个别错判(非必须)。
- 效果:力量数据从此无"未分类"黑洞。

### 个人基线为参照 (Personal Baseline, not Absolute Standard)
AI 判断合理性,**以用户本人的历史训练节奏为参照**,而非套用教科书式绝对标准(如"每周必须练腿 2 次")。
- **为何是灵魂**:用户是**非常规训练者**(撸铁 + 马拉松并行),通用健美/跑步标准本就不完全适用。合理性只能相对"你平时怎么练"来谈。
- **落地**:AI 看**滚动周期的容量/频率趋势**(如"腿的周频率是否掉了""胸的周容量是否下滑"),**不逐日点名**空缺。某天没练,只要周节奏未崩,不是问题。
- **存疑而非指责**:当空缺**偏离个人基线**(如平时周练腿 1 次却连续三周空),AI **带不确定性提出**("若为伤病/减载可忽略,若为遗漏建议补上"),而非武断判错。呼应「休整标注」的触发式设计。
> 这是最初"AI 分析合理性"那把尺的收敛:在「通用训练科学」与「个人历史基线」之间,**以个人基线为主**。

### 分析可迭代 (Iterative Analysis)
AI 分析**不是一次性静态报告**,而是"补充上下文 → 重新生成"的可迭代循环。
- **形态**:用户补标注(如休整)后,点**"重新分析"按钮**,AI 拿更新后的上下文重跑一次,生成新的「分析报告」。
- **交互边界(已收敛)**:支持**「浅追问」(第 1 档)** 就当前报告自然语言追问,但**不做会话级/跨会话对话**。
- **为何砍深度对话**:曾担心"对话变成第三个数据入口"污染数据;该顾虑已由「分析对话不产生数据」(对话只读、不回写、不缓存数据,一切以库为准)根除。故放开到浅追问即可,再深则引入发散与维护成本,demo 期不值得。

### 分析对话不产生数据 (Analysis Never Mutates Data)
AI 分析这一环,无论怎么迭代,**只读已有数据 + 读休整标注,绝不从交互中新增或修改训练数据**。
- 要改运动数据,只能走正规入口:训记镜像同步、佳明文件上传。手动记录入口仅预留,demo 阶段不开发;未来若开发,也只记录外部数据源没有覆盖的运动事实。
- **分析层与数据层严格分离**:"重新分析"之所以干净,正因它只是"拿更新后的已有数据重跑",不引入任何新数据通道。
- **对话也不缓存数据**:多轮追问时,数据**每轮从库实时重取**,绝不把数据快照冻结进对话历史。冲突来自"记住了旧数据",不来自"记住了话题"——故只丢弃前者。

#### v0.2 实现范围(浅追问落地口径)

上面"每轮从库实时重取"是针对"分析对话是否会污染/滞后于数据"的早期设想。v0.2 落地时收敛为**以报告快照为依据**,口径已与用户确认,理由与"报告是快照"原则一致:

- **就一份报告追问,依据是该报告快照**:浅追问的定位是"解释**这一份**报告",喂给 AI 的是报告固化的 `analysis_context_json` + `structured_json` + `narrative_md`,不重新读库、不重新分析。想要针对新数据的判断,走「重新分析」出新报告——这正是分析层/数据层分离的体现。
- **问答挂在报告下持久化**:浅追问问答存入 `report_followups`(挂 `report_id`),下次打开报告可见。这**不构成运动数据入口**:问答是对报告的注解,和报告本身一样是快照产物,不回写训练数据、不改目标。
- **仍是第 1 档浅追问**:只解释当前报告,不做跨报告/跨会话的自由对话;不改数据、不改目标(想改目标去「AI 目标澄清」,想补事实去「休整标注」或重新同步/上传)。
- **无 LLM 兜底**:未配 LLM Key 时追问入口置灰,报告其余部分照常查看。

> 与旧表述的关系:"每轮从库实时重取"防的是"拿旧数据冒充新判断";v0.2 用"追问明确锚定在某份快照报告上、想要新判断就重新分析"达成同一目的,且更符合报告即快照的心智。

### AI 发问原则 (AI Questioning Principle)
AI 默认应基于现有数据给出带前提和置信度的判断,而不是频繁打断用户追问。**只有当缺失信息会改变主要结论或建议方向时,才向用户发问。**

AI 可以发问的典型情况:
- 数据异常明显偏离个人基线,且无休整标注解释。
- 当前目标配置不足以判断冲突取舍(例如跑步和力量都想加,但恢复窗口不足)。
- 缺失信息会导致建议方向相反。
- 存在伤病/恢复风险但缺少解释。
- 用户主动要求深入判断。

AI 不应发问的情况:
- 普通单日空缺或个人基线内的正常波动。
- 缺失信息不影响主要结论。
- 可以通过"按当前前提判断 + 标注低置信度"表达。
- 问题已经属于外部 agent 的细方案推演层。

### AI 调用触发边界 (AI Invocation Boundary)
平台内部所有会调用大模型的操作都应由用户主动触发,或由用户确认的操作触发。demo 阶段不做后台自动分析。

- 同步训记/上传佳明文件后,**不自动调用 LLM 分析**;用户需要主动点分析。
- AI 目标澄清、AI 分析报告、浅追问、动作肌群补全、营养解释等,都属于可能消耗 LLM 的操作,应有明确触发动作。
- 对外 API 只读返回已有/已算好的存量,**不能触发平台 LLM**。

### 分析报告 (Analysis Report)
每次**主动分析/新分析**的产物是一份**可存档的结构化报告**(形态甲),而非一段聊天记录。选此形态的根本原因:**复盘是刚需**——用户的原始诉求是"两条线上都进步",看进步就必须把历次分析的关键指标存成**可比较的结构**,聊天记录做不到。
- **双层存储(关键)**:
  - **结构化层**(平台按固定字段存):分析日期、覆盖区间、各部位容量/频率、推拉比、问题标签、建议条目 → 用于**比较、画趋势、做「个人基线」**。
  - **叙述层**(存 AI 生成的自然语言):把结构化数字讲成可读的分析 → 供用户阅读。
  - 为何要结构化层:AI 是概率性的,同样数据两次措辞可能不同;**只有落成结构化字段才能稳定比较**,否则是在比较两段发散的作文。
- **报告是"这次分析的定论快照"**:一经生成即固定,供日后复盘对比。这是"便宜时先埋好的坑",与「多用户」「环境无关设计」同源。
- **历史报告不自动改**:底层训练数据、目标配置、休整标注或动作肌群映射变化后,旧报告不无痕修改;如需更新判断,通过「重新分析」生成新的报告版本。报告应记录当时使用的目标配置版本、覆盖区间、数据覆盖/同步状态摘要、主要休整标注和关键不确定项。
- **必须声明前提与不确定性**:每份报告应写明本次使用的目标配置、数据覆盖情况、主要数据缺口/目标缺口,并对关键结论给出置信度或前提说明,防止 AI 在信息不足时过度确定。

### 浅追问 (Shallow Follow-up)
AI 分析交互的多轮支持**限于第 1 档(浅追问)**:仅记住**当前这份报告**的话题线索,能接"那胸呢""再细说第二条",但不做会话级/跨会话记忆。
- **追问只出针对性的自然语言回答,不重新生成整份报告**,也**不改动结构化层的定论**(避免概率性措辞污染已定结论),更不费 token 重刷大报告。
- **"新分析"按钮**:一键清空当前话题线索、重新开始,兜住多轮聊久了发散/漂移的问题。
- 无论哪一档,数据都每轮从库实时重取(见上),故选档只决定"记不记得上一句话",不影响"数据准不准"。

### 用户档案 (User Profile)
每个用户的**静态个人信息**,为「营养素建议」提供计算依据。
- **字段(demo 期)**:身高、体重、年龄、性别。
- **不在用户档案里存目标**:目标/阶段/优先级统一归「当前目标配置」管理,避免同一用户同时存在"档案目标"与"当前目标"两套判断尺。
- **只存当前值,不追踪趋势(方案甲)**:体重变化不快,手动修正即可;**不做体重时间序列/曲线**——否则会把"填一次的资料"升级成"又一条要维护的数据线",与"砍掉饮食记录"的精神矛盾。等验证营养素建议有用,再议是否追踪体重曲线。
- **明确不收集第三层数据**(体脂率、睡眠、静息心率等):这类数据**家用手段难拿准**,喂进公式只会给出"看似精确、实则虚假"的结果,不如不要。(与「容量优先,弃用训记时间戳」同源:不可靠的输入不如不用。)
- 按 `user_id` 隔离,每用户各一份。

### 营养素建议 (Macro Recommendation)
AI 分析的一个**建议类型**(不是被砍的"饮食记录"),基于**运动量 + 用户档案**,输出当日/近期的**营养素目标**(碳水 g、蛋白 g、脂肪 g、总热量)。
- **零饮食录入**:它只**消费**运动数据与档案,**不需要用户输入吃了什么** —— 这正是它绕开饮食那块所有难点的原因(饮食记录难在"输入",营养素建议是"输出")。
- **计算输入**:①档案静态项(身高/体重/年龄/性别 → 算 BMR);②**训练消耗由平台从训记/佳明实测数据估**,不让用户手选"活动等级"(实测比估算准 —— 这是本平台胜过通用营养 App 之处);③当前目标配置(如跑步优先/力量维持/减脂/恢复等)。
- **可解释优先**:宏量营养素数字应尽量基于可解释公式/规则(用户档案 + 训练消耗 + 当前目标配置),AI 负责解释和微调,不凭空随口报数。
- **明确不做**:不生成饮食菜单,不记录实际饮食,不精确到每顿,不替代医学营养建议。
- **归类**:属于「AI 分析(云端)」的建议产物,遵循同样的分析规则(个人基线、可迭代、不产生数据等)。

### 大模型凭证 (LLM Key)
每个用户**各一个**大模型 API Key,与「训记凭证」是**两种不同信任模型**,不可混用:

| 维度 | 训记凭证 | 大模型凭证 (LLM Key) |
|---|---|---|
| 谁拥有 | 用户自己的训记账号通行证 | **平台所有者**签发 |
| 录入方向 | **用户**去训记 App 复制、填进平台 | **平台所有者在后台**给某用户绑定,用户无需看见 |
| 地址/模型 | 训记固定地址 | 平台级统一(同一模型商、同一地址) |
| 为何每用户一个 | 天然隔离各自训记数据 | **用量归属 + 可单独吊销**(见下) |

- **为何每用户一个**:平台所有者在模型商控制台按 Key 查看**每个用户的用量归属**;某 Key **用量异常时可单独吊销**,不波及其他用户。("让问题在发生点被隔离",与「同步策略」同一思路。)
- **加密存储**:与训记凭证同,数据库中加密(主密钥走环境变量,不进代码、不进库)。
- 平台调用 AI 时,用**当前登录用户的那个 LLM Key** 去调统一地址。

### 数据来源 (Data Source)
本平台的运动数据来源当前分为已接入主链路与预留能力:
- **训记 App**(力量训练)——已验证可通过其 Open API 拉取完整力量数据。见「训记镜像」。
- **佳明 / Garmin**(当前主要用于马拉松/跑步)——官方 API 企业向且已暂停,不可行;经排查,demo 期定为**手动下载 FIT 文件 → 网页端上传解析**。技术上按通用 Garmin 活动数据源设计,见「佳明活动数据源」。
- **手动录入**——仅作能力预留,demo 阶段先不开发。见「手动记录(预留)」。

### 数据主权:一条数据一个主人 (Single Ownership)
本平台的铁律,用于**从根上消除同步冲突**:任意一条运动记录**有且只有一个"主人"**,编辑权互斥,两类数据永不重叠。
- 分析时把所有来源的数据**合并呈现**,但**编辑权按来源隔离**。
> 这条约束的直接后果:本平台不需要实现双向同步与冲突合并。

### 训记镜像 (Xunji Mirror)
从训记 App 通过 Open API 同步进来的力量记录,在本平台内为**只读镜像**,且**每个用户各一份**(用各自的「训记凭证」拉取,按 `user_id` 隔离)。
- 主人是**训记**。要修改 → 回训记 App 改,再同步以训记为准。
- demo 期本平台对训记**只读不写回**,避免污染用户在训记里的真实训练数据。
- 因训记 Open API 有限频(完整读取 30s/次、轻量读取 15s/次),必须落地为**本地缓存/同步层**(同步进本地 SQLite),分析在本地数据上跑,不实时现拉。

### 同步策略 (Sync Strategy)
训记 API 按 `datestr`(天)拉取,且**不提供任何服务端变动信号**(实测训练对象无 `updated_at`/版本号/哈希)。因此无法低成本地探知"某历史天是否被改过"。基于"改历史数据每月不超过两次"这一事实,采用**两个互不相同的操作**,让罕见事件不拖累高频操作:

- **增量同步(常规操作)**:只拉「上次同步日当天 → 今天」这些天,不为历史改动做任何重扫。这是 99% 场景下点的按钮,快。
  - **边界:最近一天永远重拉一次**。因为同一天可能多次训练/多次同步(如上午练完同步、下午又练),窗口下界取「上次同步日**当天**」而非其之后,避免漏掉当天后续训练。
- **指定日重拉(补丁入口)**:用户自己知道改了哪天,手动指定该日期,平台只重拉那一天覆盖本地镜像。为每月一两次的历史改动兜底,代价只在真用时才付。

> 明确放弃:不做"滚动窗口重扫""全量定时比对"等为罕见历史改动买单的机制。首次接入为一次性全量拉取,之后全靠上述两个操作维护。
- 同步为**手动触发**(用户点"从训记同步"按钮),demo 期不做后台定时同步。
- **首次全量、之后增量**:第一次拉取历史数据量较大,此后每次只拉新增/变动的天,数据量小、耗时短。

### 训记凭证 (Xunji Credential / API Key)
每个用户在训记 App 申请、绑定到本平台的 Open API Key(形如 `xjllm_...`),是**该用户训记账号的通行证**——持有它即可读/写其全部训练数据。
- **每用户各一份**,不是平台级全局配置。平台用**各用户各自的 Key** 拉**各自的**训记数据 → 数据隔离由此天然成立(A 的 Key 物理上拉不到 B 的数据)。
- **安全等级:等同训记账号通行证**(可读、且 API 支持写回全部训练数据)。因此:
  - **必须加密存储**(demo 期至少对称加密,主密钥走环境变量,不进代码、不进库明文),脱库时不能被直接读走。
  - **绝不写入日志、不回显给任何第三方**。
- **绑定动作的开放时机**:让所有者本人以外的用户在自己手机上**输入训记 Key**,推迟到方案 C(HTTPS)之后再开放——见「朋友接入门」。demo 期仅所有者本人在可信环境下使用一个凭证。

### 手动记录(预留) (Manual Entry, Reserved)
用户在本平台手动录入运动记录的能力,**demo 阶段先不开发,仅预留概念位置**。

原因:手动记录模板尚未想清,若过早开发,容易滑成万能日记本,与「休整标注」「目标配置」「分析追问」等模块混淆,污染数据边界。

预留边界:
- 未来若开发,只应记录**训记和佳明没有覆盖的运动事实**(如未带设备的运动、其他运动类型、康复/恢复性运动等)。
- 不用于记录训练目标(归「当前目标配置」)、休整原因(归「休整标注」)、分析补充上下文(通过标注后重新分析)、饮食(已移出范围)、体重趋势(用户档案只存当前值)。
- 主人是**本平台**。手动记录只能补外部数据源没有的数据,不能覆盖或改写训记镜像/佳明上传数据。

### 佳明活动数据源 (Garmin Activity Data Source)
当前 demo 主要服务跑步/马拉松,但技术上按**通用 Garmin 活动数据源**设计,而不是跑步专用入口。FIT 文件可能来自跑步、跑步机、体育场/操场跑、越野跑、骑行、游泳、徒步等活动;解析层必须保持运动类型扩展能力。

经多轮排查,佳明各接入路径的结论:
- **官方 Garmin Connect API**:企业向(拒个人申请)+ 程序当前暂停 → 不可行。
- **服务器模拟登录(python-garminconnect)**:需存佳明**账号密码**(泄露=整个账号沦陷),且非官方、佳明改风控即失效 → demo 期不采用。
- **Strava 中转(OAuth)**:安全(不碰密码、令牌可撤销)但需所有者约 $12/月 Strava 会员 → 暂不采用。
- **手动下载文件(demo 期采用)**:用户自行从佳明导出**「原始格式」= 一个 `.zip`,内含一个 `.fit` 文件**。`.fit` 是佳明原生二进制,数据最全(逐秒心率、逐秒配速、GPS、功率、步频),正是马拉松分析所需。

**手动文件路的两点约定:**
- **主链路(喂分析)**:手机下载 → 直接上传网页端平台。平台负责**解压 zip → 解析 FIT → 查重 → 分析**。
- **查重(平台自带、幂等)**:平台以 FIT 内的**活动开始时间戳**(必要时结合活动 ID)为唯一键去重 —— 同一次跑步无论从手机/PC、传几次,平台只认一条。这是「数据主权:一条数据一个主人」在文件来源上的落地。
- **解析依赖**:平台与 PC 工具都需引入一个 **FIT 解析库**(二进制格式,不能直接当文本读)。属实现细节。

**FIT 解析分层:**
1. **通用活动解析层**:先解析所有 Garmin 活动共有事实,不把入口写死为跑步。字段包括活动唯一键、开始/结束时间、elapsed time、timer time、sport/sub_sport、活动名称、距离、速度/配速、心率、功率、步频/踏频、海拔、气温、消耗热量、GPS 是否存在、lap/分段、字段覆盖情况、原始文件引用。
2. **运动类型标准化层**:将 Garmin `sport/sub_sport` 映射为平台内部 `activity_family` 与 `activity_variant`。如 running / road_run、running / treadmill_run、running / track_run、running / trail_run、cycling、swimming、hiking、unknown。
3. **运动类型专属解释层**:不同运动类型进入分析时使用不同解释器。demo 阶段只对 running family 做完整分析;骑行、游泳等未来通过专属解释器扩展。

**demo 非跑步活动处理:**允许上传非跑步 FIT,但 demo 阶段只做通用解析、查重、存档、类型标记和元数据展示;暂不进入双线负荷矩阵、不做专项 AI 建议、不作为目标判断核心依据。越野跑作为 running family 的变体预留,未来可比骑行/游泳更早纳入跑步解释器。

**跑步分析定位:**平台不复刻 Garmin,不做一个“小 Garmin”。Garmin 负责专业跑步指标展示;本平台负责把跑步活动转化为双线负荷的一部分,重点服务于过量判断、下肢冲突、恢复压力、营养需求和目标取舍。

**FIT demo 期目标解析字段:**活动时间、活动类型、距离、时长/移动时长、配速/速度、心率、分段/lap 或配速波动、海拔爬升/下降、**步频、功率、气温**、消耗热量。不同 FIT 不保证都有全部字段;平台应有则解析/存储/纳入分析,缺失则记录缺失并在报告中声明影响。若 FIT 缺少气温,可在后续版本用户外活动的 GPS 起点 + 活动时间从第三方历史天气服务补全气温/湿度等环境上下文;该能力不作为 demo 内容,优先级不高。

**跑步活动归类分两层:**
- **训练类型**:轻松跑、长距离跑、节奏跑、间歇/速度课、恢复跑、比赛/测试、未知/混合。
- **跑步场景**:户外路跑、体育场/操场跑、跑步机、未知。

跑步机活动计入有氧负荷、下肢负荷、全身恢复负荷和营养需求,但其距离/配速/海拔/气温等字段的解释需按室内场景调整置信度;海拔通常不用于跑步机负荷判断。体育场/操场跑有 GPS,但路径范围小、海拔通常无意义;它正常计入跑步负荷和下肢负荷,但解释配速波动、GPS 轨迹和爬升时应按场地跑处理。步频、功率、气温用于更准确地解释强度、疲劳、环境影响和双线冲突,但 demo 阶段不做高级跑姿诊断、功率训练处方、热应激医学模型、地图轨迹展示或 Garmin 训练状态复刻。

**FIT 实测结论(2026-07-08):**已用 3 个真实 Garmin 导出 zip 验证:一个 zip 内含一个 `*_ACTIVITY.fit`;跑步机、户外路跑、体育场/操场跑均可解析。实测可取得活动时间、距离、elapsed time、timer time、配速/速度、心率、步频、功率、消耗热量、lap/分段;户外路跑/体育场有 GPS,户外路跑有海拔起伏,跑步机无 GPS/海拔;3 个样本均未提供气温,故气温必须按"目标字段但允许缺失"处理。FIT 解析链路技术可行。

### PC 重命名工具 (PC Rename Utility)
一个**独立于平台**的 PC 端轻量小工具,仅服务于用户**本地文件管理**,与网页端平台**完全解耦、不共享状态**。
- **唯一职责**:解压/读取下载的 FIT,取活动开始时间,把乱码文件名(如 `9A3F7C2E.zip`)**按运动时间重命名**(如 `2026-07-06_晨跑.fit`),方便用户在磁盘上肉眼核对、**防自己下重/下漏**。
- **明确不做**:不联网、不连平台、不维护"下过哪些"的账本。它不是平台的一部分,平台没有它也完整运转。
> 边界澄清:平台的查重(幂等去重)与这个工具的"防下漏"是**两回事**——前者管"进平台的数据不重复",后者纯粹是用户整理本地磁盘文件的私人便利。二者不共享账本,故不会互相打架。

### 用户 (User)
使用本平台的自然人。范围:平台所有者本人 + 朋友,共少数几人,分散在不同地点、各自用手机访问。
- **本平台从一开始就按多用户建模**:数据一律按 `user_id` 隔离,每个用户各绑各的训记凭证、只见自己的数据。
- **"多用户建模"是架构决定,"开放朋友接入"是运营开关**——两者分离。demo 期开关**关着**(见「朋友接入门」),仅所有者本人使用,但代码与数据模型不因此退化为单用户。
> 便宜时就把贵的坑填了:加 `user_id` 字段几乎零成本;而事后给历史数据补 `user_id`、重构所有查询是伤筋动骨的返工。此原则与「环境无关设计」同源。
> 注:后续功能讨论中若出现"账号""成员"等词,需回到此处对齐,避免与"用户"混用。

### 朋友接入门 (Friend Onboarding Gate)
一个明确的运营开关:**"是否允许所有者本人以外的用户注册并填入自己的训记 Key"**。
- **前提条件:方案 C(正规 HTTPS)就位**。理由:让他人在明文(方案 A)信道上输入等同账号通行证的训记 Key,等于让凭证裸奔被嗅探——这个责任 demo 期不背。
- demo 期此门**关闭**:仅所有者本人使用(自己填 Key,风险自担、在可信网络下操作)。
> 与「决策门」区别:决策门决定"要不要上线投入";朋友接入门决定"多用户对外开放的时机",且被 HTTPS 硬前提卡住。

### 方案 A(免备案 · IP 直连)
部署形态之一。通过 `http://<公网IP>:<端口>` 直接访问,**不绑域名、不备案**。
- 用途:**开发/调试阶段**使用。
- 代价:明文传输(无正规 HTTPS)、地址不友好。
- 适用前提:小范围(本人+家人)试用,风险可接受。

### 方案 C(备案 · 域名 · 正规 HTTPS)
部署形态之一。绑定已备案域名 + Let's Encrypt 正规 HTTPS。
- 用途:**上线阶段**使用(仅当通过"决策门")。
- 前提:大陆节点服务器必须完成 ICP 备案(域名需 `.com`/`.cn` 等可备案后缀)。

### 决策门 (Decision Gate)
一个明确的判断节点:demo 完成并经本人+家人试用后,再决定**是否值得投入**上线(买域名、备案等高成本动作)。
- 通过 → 进入方案 C。
- 不通过 → 砍掉,无沉没成本(未预付域名/备案)。
> 核心意图:高成本动作(备案)一律推迟到决策门之后,不拖累 demo 效率。

### 环境无关设计 (Environment-agnostic Design)
一条贯穿始终的工程约束:所有"环境相关项"(API 地址、协议 http/https、Cookie 的 Secure 策略、CORS/来源白名单)必须走**配置/环境变量**,代码内**不写死**。
- 目的:保证方案 A → 方案 C 的切换只需"改配置 + 装证书",而非改代码返工。

### 平台对外 API(已实现) (Outbound Open API)
本平台除了**消费**训记/佳明的数据,自身也**对外提供一套 Open API**,让外部 AI agent 能像调训记那样调本平台——从而在平台之外组合出更丰富的玩法。**平台由此获得双重身份:既是上游数据源的消费者,又是别人的上游数据源。**

v0.4.0 已实现 `/api/v1/*` 共 9 个只读 GET 端点 (meta/context/days/goals/reports/muscle-map/rest-notes)。`skills/strength-running-readonly-api/` 供外部 Agent/LLM 通过 curl 只读消费。详见 ADR 0004 和 [docs/design/api-design.md](docs/design/api-design.md) §12。

**鉴权模型(与训记同构)**:
- 每个用户可在平台**签发多个"平台 API Key"**(`srda_` 前缀),外部 agent 持此 Key 调用。
- Key **绑定某个 `user_id`**,agent 只能读该用户自己的数据,越权不了(天然复用「数据主权:一条数据一个主人」与多用户隔离)。
- **hash-only 存储**:数据库 `api_keys` 表仅存 sha256 哈希;明文**仅签发时一次性显示**,之后不可再现。可单独吊销。与可解密 AES-GCM 加密的训记/LLM Key 是不同的存储模型。

**功能边界(只读"存量",不碰"增量")**:
- ✅ **只读暴露平台已拥有/已算好的东西**:原始训练数据(未加工明细,按日期/区间)、加工上下文(聚合+肌群的结构化数据,即喂 LLM 的那份)、**当前/历史目标配置**、动作→肌群映射表、历史分析报告、休整标注、元数据(有哪些数据/覆盖日期/上次同步时间)。
- ❌ **排除一切"产生增量/花额度/改状态"的操作**:写入(包括外部 agent 写回训练计划、目标、标注、报告等)、**让 agent 触发分析**(会消耗 LLM 额度、生成新报告 → **连锁调用致费用不可控**,明确不做)。
- 设计原则一句话:**读存量(零边际成本),不碰增量(费用/状态风险)。**

**部署架构**:
- demo 期已摆正两层解耦:
  1. **上下文生产独立成层**:AI 分析与 API 共用同一 `build_context` 生产者,不塞进"分析按钮"的代码路径。
  2. **数据访问按 `user_id` 解耦,不绑死网页 session**:底层只认当前 `user_id`;其来源(网页 cookie / API Key)由上层解析。
- HTTP 路由与 Key 管理界面作为"加法"叠加上去,未伤及已有代码。
